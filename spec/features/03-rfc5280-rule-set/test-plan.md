# Test Plan: RFC 5280 Rule Set

## Scope

Verify each of the six RFC 5280 lints flags its targeted violation, passes a clean cert,
and scopes correctly via `applies()` (CA-only lints are `NotApplicable` on leaves).

## Conventions

Per `.claude/rules/rust-testing-core.md`: SIFER, `.unwrap()`/`.unwrap_err()`, behaviour-
focused tests grouped per lint in nested modules. One fixture per lint that violates
exactly that rule; `good.pem` passes all six.

## Fixtures (`testdata/`)

- `rfc5280_version_not_v3.pem`, `rfc5280_serial_number_zero.pem`,
  `rfc5280_validity_inverted.pem`, `rfc5280_ca_bc_not_critical.pem`,
  `rfc5280_ca_missing_keycertsign.pem`, `rfc5280_empty_subject_no_san.pem`.
- Reuse `good.pem` (passes everything). Regenerated by `testdata/generate.sh`.

## Integration Tests (`crates/linter/tests/rfc5280.rs`)

- Per lint: fixture → `check` returns ≥1 `Error` with a relevant message; `good.pem` →
  empty findings.
- CA-only lints (`basic_constraints_critical_on_ca`, `key_usage_present_when_ca`) report
  `NotApplicable` on a leaf.
- `version_is_v3` is `NotApplicable` when no extensions are present.
- `san_present_if_subject_empty` is `NotApplicable` when subject is non-empty.
- Full `default_registry()` over `good.pem` → no Error/Fatal from the RFC 5280 source.

## Edge Cases

- Serial: zero, negative (high bit set), and >20 octets each flagged.
- A lint that legitimately fails for two reasons returns multiple findings (e.g. SAN
  present but not critical alongside any second issue).
- Malformed serial / missing extension fields handled without panic.

## Verification Commands

```
cargo test
cargo clippy --all-targets -- -D warnings
cargo fmt --check
bash testdata/generate.sh   # fixtures regenerate cleanly
```

## Exit Criteria

All six lints validated against dedicated fixtures and `good.pem`; applicability correct;
verification commands pass.
