# Test Plan: CA/Browser Forum BR Rule Set

## Scope

Verify the four broad-scoped BR lints (`cabf_br_validity_max_398_days`, `cabf_br_cn_in_san`,
`cabf_br_no_internal_names_or_reserved_ip`, `cabf_br_ext_key_usage_server_auth_present`), the SAN/EKU
accessors, and the reserved-IP/internal-name classifier. **Broad scoping:** every non-CA leaf is in
scope for all four; CA certs are `NotApplicable`. NOT EKU-gated.

This feature ALSO regenerates every shared/cross-feature non-CA leaf fixture (features 03/04) to be
BR-compliant-except-target, and updates the affected test files. That regeneration is part of THIS
test plan (tester task 04 + developer task 05 for the cert.rs unit test).

## Conventions

Per `.claude/rules/rust-testing-core.md`: SIFER, `.unwrap()`/`.unwrap_err()`, behaviour-focused tests
grouped per lint.

## ⚠️ Time-Fragility (must be understood by anyone touching these fixtures)

BR-compliant leaves use a currently-valid ≤398-day window `2026-06-01 → 2027-06-01`. These fixtures
**expire on 2027-06-01**, after which `hygiene_not_expired` fires on every leaf and the isolation
tests fail wholesale. **Regenerate annually (slide the window forward) before 2027-06-01.** The
`cabf_br_validity_400_days` fixture (`2026-06-01 → 2027-07-06`) has a slightly later horizon.
`generate.sh` documents this loudly; test module docs should reference it so a flood of `not_expired`
failures is diagnosable.

## Fixtures (`testdata/`)

### Regenerated non-CA leaf fixtures (BR-compliant except their one target)

All gain `serverAuth` EKU, a SAN whose dNSName entries include the subject CN, no internal/reserved
SAN entries, and a `BR_OK` (2026-06-01 → 2027-06-01, 365d) window — except where the target requires
deviating from exactly one of those:

- `good.pem` — fully clean BR-compliant leaf (DNS:good.example = CN, serverAuth, 365d, RSA-2048/
  SHA-256). Passes the entire 14-lint registry.
- `expired.pem` — past ≤398d window (`2024-01-01 → 2024-06-01`, 151d) + serverAuth + SAN-with-CN;
  violates ONLY `hygiene_not_expired`. **`notAfter` Unix seconds = `1_717_200_000`** (drives the
  `EXPIRED_NOT_AFTER` constants).
- `rfc5280_serial_number_zero.pem` — serial 0; else BR-compliant.
- `rfc5280_validity_inverted.pem` — zero-length window at a FUTURE instant (e.g.
  `2027-01-01 = 2027-01-01`); serverAuth + SAN-with-CN added; violates ONLY
  `rfc5280_validity_not_after_after_not_before`.
- `rfc5280_empty_subject_no_san.pem` — empty subject, NO SAN (target), but serverAuth EKU + BR_OK
  window ADDED so it isolates ONLY `rfc5280_san_present_if_subject_empty` (no CN ⇒ cn_in_san silent;
  no SAN ⇒ internal-name lint silent; serverAuth present ⇒ EKU lint silent).
- `rfc5280_version_not_v3.pem` — v3-byte patched to v1; else BR-compliant (serverAuth + SAN-with-CN +
  BR_OK).
- `hygiene_sha1_signature.pem`, `hygiene_rsa_1024.pem`, `hygiene_ecdsa_bad_curve.pem` — each its one
  hygiene violation; all gain serverAuth + SAN-with-CN + BR_OK.

### New BR per-lint fixtures

- `cabf_br_validity_400_days.pem` — 400d currently-valid window (`2026-06-01 → 2027-07-06`),
  serverAuth + SAN-with-CN; violates ONLY `cabf_br_validity_max_398_days`.
- `cabf_br_cn_not_in_san.pem` — serverAuth + a SAN that OMITS the CN; violates ONLY `cabf_br_cn_in_san`.
- `cabf_br_internal_san.pem` — serverAuth + CN-in-SAN (public) PLUS an internal name AND a reserved
  IP (e.g. DNS:internal.local + IP:10.0.0.1); violates ONLY `cabf_br_no_internal_names_or_reserved_ip`
  (and yields MULTIPLE findings).
- `cabf_br_missing_serverauth.pem` — SAN-with-CN + EKU present WITHOUT serverAuth (e.g. clientAuth
  only); violates ONLY `cabf_br_ext_key_usage_server_auth_present`.

### CA fixtures (unchanged — BR N/A)

- `rfc5280_ca_bc_not_critical.pem`, `rfc5280_ca_missing_keycertsign.pem`.

All regenerated by `testdata/generate.sh`.

## Unit Tests (`reserved.rs`, owned by developer task 01)

- `is_reserved_ip`: true for 10.0.0.1, 127.0.0.1, 169.254.x, ::1, fe80::, 192.0.2.x; false for a
  public IP.
- `is_internal_name`: true for `foo.internal`, `host.local`, single-label `localhost`; false for
  `example.com`.

  `good_cert_subject_is_not_empty`, `good_cert_serial_*`, all `spki_accessors` tests (good.pem stays

## Integration Tests (`crates/linter/tests/cabf_br.rs`)

- Per lint: its fixture flagged with a descriptive message (names the offending CN / SAN entry /
  duration); `good.pem` passes.
- `cabf_br_cn_in_san`: a CN absent from SAN is flagged (message names the CN); a CN-only subject with
  matching SAN passes; a subject with NO CN yields no finding.
- `cabf_br_no_internal_names_or_reserved_ip`: multiple bad SAN entries → multiple findings (one per
  offending entry).
- `cabf_br_validity_max_398_days`: 400-day cert flagged (message names duration); ≤398 passes;
  boundary 398 passes / 399 flagged.
- `cabf_br_ext_key_usage_server_auth_present`: clientAuth-only fixture flagged; serverAuth present
  passes.
- All four BR lints `NotApplicable` on a CA cert (use `rfc5280_ca_bc_not_critical.pem`).

## Edge Cases

- Mixed SAN: one public + one internal name → only the internal entry flagged.
- Validity exactly 398 days → passes; 399 → flagged (boundary).
- IPv6 reserved ranges classified correctly (::1, fe80::).
- CN matching SAN case-insensitively for dNSName (document chosen policy and test it).
- Subject with no CN → `cabf_br_cn_in_san` silent.
